# Output Contract

Use this reference when reporting results from browsing or web research.

## Goal

Make the answer easy to trust.
Do not force the user to interpret unstable intermediate browsing states.

## Default output order

### 1. Final answer first
Lead with the stable conclusion:
- what you found
- whether the page/source exists
- which source is the best evidence
- what changed / what is true now

### 2. Evidence second
Then give supporting details:
- direct links
- page titles
- quoted or summarized facts
- screenshots only when they add value

### 3. Follow-up last
Only after the main answer is clear:
- optional next checks
- extra candidate sources
- additional screenshots

## Stable-first rule

Do not present the first partial page state as the final answer.
Wait until the page state is good enough to support a stable conclusion.

Examples of unstable states:
- only seeing a landing page when the target content has not been opened
- only seeing search suggestions without opening the real target page
- only reading a snippet when the full page is accessible
- seeing an incomplete render and assuming the page has no content

## Screenshot rule

Use screenshots when they materially improve trust or understanding:
- UI bug reports
- visual confirmation of a page state
- proof that a post/article/page exists or looks a certain way
- before/after comparisons

Do not attach screenshots by default when text evidence is enough.

## Language rule

Prefer plain, direct reporting.
State uncertainty explicitly when evidence is partial.

Examples:
- “我确认这是该账号的主页，当前可见最新一条内容是…”
- “我只确认到搜索结果页，还没进入目标文章，因此这个结论还不稳定。”
- “目前只能找到二手来源，没看到官方原文。”

## Comparison tasks

For comparisons, keep the final structure stable:
- summary table or bullets first
- per-source detail second
- screenshots only if the differences are visual and important


## Discovery surface warning

If you are still on a search/discovery surface, say so.
Do not report a final answer as if the target page itself has already been verified.
